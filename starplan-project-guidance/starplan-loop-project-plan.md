# 星程 StarPlan Loop 项目总体计划

日期：2026-07-18  
状态：方案讨论稿，尚未进入开发

## 1. 参赛方向

参赛赛道为挑战杯“揭榜挂帅”阿里云榜题：

**赛道三方向三：星语·面向 AI 的天文实训**

赛题要求参赛团队把真实天文工作或科普实践中常用、但尚未很好接入 AI 的任务环节，改造成智能体可以稳定调用的天文 Skills。作品应具备以下特征：

- 可运行
- 可检查
- 可复现
- 可复用
- 有明确的输入、输出、依赖、触发条件和失败处理
- 实际使用阿里云 Qwen 系列模型及相关产品能力

项目不做泛用天文聊天助手，也不以搭建大而全的天文网站为目标。

## 2. 修改后的项目定位

### 项目名称

**星程 StarPlan Loop：面向 AI 的校园天文观测与科普实训闭环 Skills 包**

### 一句话简介

StarPlan Loop 将目标解析、可观测性计算、科普活动设计、实际观测记录和偏差复盘封装为 Qwen 智能体可调用的 AI Ready Skills，让一次校园观测活动能够被计划、执行、检查并持续改进。

### 核心使用场景

主要服务对象为：

- 高校天文社团及其新成员
- 中小学天文科普教师
- 校园公众观测活动组织者
- 本科生天文实训课程

第一使用场景确定为：

> 天文社团希望在指定地点和日期组织一次面向新成员的观测活动，需要判断目标是否适合观测、选择时间、准备讲解和设备，并在活动结束后复盘实际结果。

## 3. 项目要解决的问题

传统观测活动需要组织者在星表、星历、观测计划软件、星图软件和文档工具之间切换。新手容易遗漏时区、目标名称歧义、低高度、月光和设备条件；最终往往只留下截图和活动文档，难以追溯计划依据，也无法系统总结失败原因。

StarPlan Loop 将流程改造为：

```text
自然语言活动需求
-> Qwen 解析为结构化任务
-> 天文工具完成确定性计算
-> 生成观测计划和科普活动包
-> 组织者执行活动并填写观测日志
-> 系统对比计划与实际结果
-> 解释偏差并生成下一次修订计划
```

核心原则：

```text
工具算，模型讲，报告验，人员确认，日志促改进
```

Qwen 不直接生成高度角、方位角、日落时间、月亮位置和行星坐标等数值。所有关键数值必须来自确定性天文工具，并保留来源和中间结果。

## 4. 项目目标

### 科学目标

- 对目标、地点、日期和设备进行规范化处理。
- 可靠计算高度角、方位角、airmass、暮光和月光影响。
- 给出带依据的推荐窗口、风险说明和备选方案。
- 让科普文案严格依据计算结果和经过审核的事实卡生成。

### 技术目标

- 交付可以单独调用、也可以由总控串联的 Skills 包。
- 为每个 Skill 定义输入、输出、触发条件、依赖和失败处理。
- 使用 Qwen/阿里云百炼完成自然语言解析、工具调用编排和科普表达。
- 保存模型版本、工具版本、调用链和关键中间结果。
- 在无在线天文服务时，核心观测计算仍能复现。

### 应用目标

- 让天文社团新成员能理解推荐时间是如何产生的。
- 降低组织校园观测活动的准备成本和专业门槛。
- 在活动失败或效果不佳时，给出可检查的偏差归因和下一轮改进建议。
- 形成可以用于社团培训、课堂实训和公众科普的标准活动记录。

## 5. 核心 Skills 设计

首期收敛为 4 个核心 Skill 和 1 个总控入口。星图和图表作为计算结果的展示层，不单独拆成 Skill；太阳系星历作为后续扩展，避免首期范围过大。

| Skill | 主要职责 | 典型输入 | 主要输出 | 关键失败处理 |
|---|---|---|---|---|
| `target_resolve` | 将中文名、英文名或别名解析为标准目标 | 目标名称、可选目标类型 | 标准名称、类型、坐标、来源、置信度 | 名称歧义时列出候选并要求人工确认 |
| `observability_plan` | 计算目标是否可观测并生成计划 | 目标坐标、地点、日期、设备约束 | 可见窗口、高度/方位、airmass、暮光和月光风险、备选时段 | 地点、时区或日期不完整时停止计算并返回缺失项 |
| `outreach_pack` | 把计算结果转成适合目标受众的科普活动包 | 事实卡、计算结果、受众、设备 | 活动流程、讲解词、设备清单、安全提示、人工核对项 | 事实不足时不补写数值，只标记待确认内容 |
| `observation_review` | 对比计划和实际日志并修订下一轮计划 | 原计划、实际时间、结果、云量、设备、备注 | 偏差分类、证据、改进建议、修订后的下一次计划 | 信息不足时区分“无法判断”和“可能原因” |

总控入口：

```text
starplan.run
```

总控负责识别用户意图、依次调用必要 Skill、保存运行目录，并把机器可读结果和人类可读报告一起返回。

## 6. 数据流与输出规范

### 统一输入示例

```json
{
  "target": "M31",
  "location": "北京某高校",
  "date_range": ["2026-10-09", "2026-10-11"],
  "audience": "天文社新成员",
  "equipment": "双筒望远镜",
  "goal": "校园科普观测"
}
```

### 统一运行目录

每次运行生成独立目录，建议包括：

```text
runs/case_xxx/
  input.json
  resolved_target.json
  calculation_manifest.json
  observability.csv
  visibility_curve.png
  plan.json
  outreach_pack.md
  observation_log.csv
  review_report.md
  revised_plan.json
  validation_report.md
  model_call_log.jsonl
```

其中 `calculation_manifest.json` 是本项目的重要证据文件，记录：

- 输入地点、时区和时间范围
- 目标来源和坐标
- 使用的天文工具及版本
- 主要约束规则
- 中间计算结果文件
- 人工修改和确认记录

## 7. MVP 范围

### 第一阶段必须完成

- 常用目标别名解析和名称歧义处理。
- 固定恒星及深空目标的本地可观测性计算。
- 高度角、airmass、暮光和月光影响分析。
- 带理由的推荐窗口、风险和备选时间。
- 基于事实卡的 Qwen 科普活动包。
- CSV 观测日志导入、偏差归因和修订计划。
- 一个轻量网页调用面板或等价演示入口。
- 3 组完整案例、自动测试和复现文档。

### 第二阶段再考虑

- 行星和太阳系目标的权威星历接入。
- Stellarium 脚本或 Aladin Lite 交互星图。
- 校园地平线遮挡模型。
- 更多天象模板和多语言科普材料。

### 暂不进入范围

- 望远镜硬件控制
- 实时天气预报系统
- 账户、社区和长期内容平台
- 覆盖所有类型天体和天象
- 模型微调、多智能体系统和复杂 3D 前端

## 8. 三类典型案例

### 案例一：正常可观测活动

北京高校天文社计划组织 M31 新生观测夜。系统生成目标信息、推荐时段、高度曲线、设备建议和讲解包，人工确认后形成活动计划。

### 案例二：不适合观测及备选方案

用户输入的目标在指定时间高度过低、受月光影响或处于暮光阶段。系统不能只返回“不推荐”，还要展示触发的规则、计算依据和可替换的时间或目标。

### 案例三：实际活动复盘

导入一份包含迟到、云量增加、器材准备不足或目标选取不当的观测日志。系统区分可计算因素与人工记录因素，生成偏差报告并修订下一次活动计划。

若核心流程提前稳定，可增加土星观测作为太阳系扩展示例，但不得影响上述三类案例的完整性。

## 9. 验证与评测计划

### 科学准确性

- 使用成熟天文软件或公开权威结果对固定案例进行交叉校验。
- 检查目标坐标、时区、太阳高度、目标高度和推荐窗口。
- 保存参考结果、差异和人工确认结论。

### 模型可靠性

- 检查科普稿中的所有时间、角度和天体事实是否能追溯到事实卡或工具输出。
- 测试 Qwen 遇到缺失数据时是否会主动标记，而不是补写不存在的结果。
- 比较不使用工具调用与使用 StarPlan Skills 时的幻觉率和可追溯性。

### 闭环有效性

- 设计至少 3 类观测偏差：时间偏差、环境影响、设备或操作问题。
- 检查复盘结果是否引用了计划和日志证据。
- 检查下一次计划是否真实修改了相应参数，而不是重复原建议。

### 可复现性

- 在新的运行环境按 README 完成安装和固定案例复跑。
- 确保相同版本、相同输入能产生一致的计算结果。
- API Key 仅影响 Qwen 表达层，核心计算和示例结果应有离线复现方式。

## 10. 最终成果形式

### 核心作品

1. **StarPlan Loop Skills 包**
   - Skills 定义文件
   - Python 源码和统一调用入口
   - 安装、依赖、调用示例和失败说明
   - 阿里云百炼/Qwen 接入配置说明

2. **轻量演示应用**
   - 输入目标、地点、日期、受众和设备
   - 展示目标解析、中间计算、推荐窗口和风险
   - 导出活动包
   - 导入日志并展示复盘和修订结果

3. **复现与评测材料**
   - 3 组完整运行记录
   - 固定评测集和自动测试
   - 科学交叉校验记录
   - Qwen 调用日志和模型约束说明
   - 开源工具来源、许可证和改造说明

### 赛事提交材料

- 不超过 20 页的 PPT/PDF
- 应用技术报告
- 完整源码和依赖说明
- Skills 清单与调用流程图
- 3 类任务的输入、中间结果、最终输出和人工校验记录
- 10 分钟以内演示视频
- 可选的在线或本地交互入口

### 现场演示主线

```text
输入校园观测需求
-> 展示 Qwen 的结构化解析
-> 展示确定性计算及来源
-> 生成计划和科普活动包
-> 导入一次失败或效果不佳的活动日志
-> 展示偏差依据和修订后的下一次计划
```

## 11. 五人分工建议

| 成员 | 负责方向 | 主要交付 |
|---|---|---|
| A | 项目统筹与科学校验 | 案例、规则、参考结果、验证报告、总体叙事 |
| B | 天文计算内核 | `observability_plan`、计算文件、曲线和测试 |
| C | 目标数据与结果展示 | `target_resolve`、目标目录、图表、可选外部工具接入 |
| D | Qwen 与 Skills 编排 | 总控入口、Schema、百炼调用、`outreach_pack`、审计日志 |
| E | 复盘与交付展示 | `observation_review`、演示页、README、PPT 和视频 |

所有成员共同维护 3 个固定案例。输入输出 Schema 变更必须经过全组确认。

## 12. 阶段安排

| 阶段 | 主要目标 | 阶段验收 |
|---|---|---|
| 第 1 周 | 冻结范围、Schema、案例和验证规则 | Skills 清单、输入样例、输出目录、测试基线 |
| 第 2 周 | 跑通目标解析和本地可观测性计算 | M31 案例能生成结构化计算结果和验证报告 |
| 第 3 周 | 接入 Qwen 编排和科普活动包 | 模型只使用事实卡，调用过程可追踪 |
| 第 4 周 | 完成观测日志与复盘闭环 | 能从计划和日志生成有依据的修订计划 |
| 第 5 周 | 完成演示入口和 3 类案例 | 新环境可复跑，失败场景可解释 |
| 第 6 周 | 打磨评测、报告和视频 | 20 页材料、演示视频、源码和复现包齐全 |

## 13. 项目成功标准

项目达到以下条件才视为完成：

- 不是聊天演示，而是可以直接调用的 Skills 包。
- 至少一个核心案例从计划、执行记录到修订计划完整闭环。
- 至少三类任务具有可复现运行记录。
- 所有关键天文数值都能追溯到工具输出。
- Qwen 的作用、边界和调用证据清楚。
- 评委可以在有限时间内看懂任务价值、运行过程和创新点。
- 团队成员能解释每个模块的输入、输出、依赖、失败处理和验证方式。

## 14. 当前需要团队确认的事项

- 第一批支持的目标目录和地点范围。
- 统一的观测日志字段。
- 3 个最终演示案例的具体日期和场景。
- 采用的阿里云百炼模型及调用方式。
- 交叉验证所使用的参考工具和误差标准。
- 比赛最新提交入口、截止日期和材料格式。
