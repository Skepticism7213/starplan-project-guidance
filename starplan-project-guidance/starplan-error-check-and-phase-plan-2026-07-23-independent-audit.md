# StarPlan Loop 独立错误排查与阶段安排

日期：2026-07-23
审查范围：`starplan-loop-project-plan.md`、`starplan-error-check-and-phase-plan-2026-07-23.md`、`StarPlan/` 全部受版本控制内容，以及当前本地运行入口
审查性质：只读代码审查与运行验证；本轮未修改 StarPlan 业务代码和数据

---

## 一、结论摘要

当前版本已经具备可运行的雏形：3 个结构化案例均能退出码 0 运行，Qwen 单轮、JSON、自然语言解析和 function calling 均能实际调用；本地 Schema、编译、Layer 1 和 8 项数字幻觉测试也通过。

但是，`2026-07-23` 报告中的“第 3 周核心已完成”“0 个新增 CRITICAL”“科学交叉校验已超额完成且可直接作为准确性证据”不能按当前仓库状态验收。本轮发现 **5 个 CRITICAL、11 个 WARNING、7 个 INFO**。其中最重要的问题是：

1. 月球与目标的角距离在不同坐标系之间直接计算，警告被屏蔽而不是修复；M31 实测样点从错误的约 `33.6°` 变为同一 AltAz 坐标系下的约 `105.6°`。
2. M42 明确不可观测，但仍生成“今晚一起观测 M42”的活动包。
3. Qwen function calling 的最终答复未通过数值溯源检查，仍原样返回用户；本次真实调用检出 15 个无法按现有规则溯源的数值。
4. 目标歧义只打印提示，不会暂停等待人工确认。
5. `calculation_manifest.json` 的 `validation_status` 被硬编码为 `passed`，并会在模板模式下仍声称使用 Qwen，不能作为可信证据链。

因此，当前正确定位应为：**第 2 周主体可运行，第 3 周已接通但未达到验收，第 4 周规则版复盘为早期原型；尚不能称为领先计划约 1 周。**

## 六、阶段计划与验收标准

### P0 修复批次（建议 1-3 天）

1. **修复月球坐标链**：目标与月球统一到同一时刻、地点和坐标系；移除警告屏蔽；至少用 astroplan/独立公式交叉验证 M31、M42 和一个近月目标。
   验收：角距误差小于 0.1°；月球在地平线下不触发月光淘汰；所有相关测试通过。
2. **把 Qwen 防线改为 fail closed**：工具结果先形成结构化“允许陈述”，Qwen 只能改写；校验失败时删掉/重生成对应句，不得原样返回。
   验收：真实 chat 中 `passed=true` 才返回正式结果；数字舍入有明确容差；文字事实也有来源 ID。
3. **修复失败场景和人工确认**：不可观测时生成替代方案包；任何 `requires_confirmation=true` 必须停止并返回候选。
   验收：M42 不再声称“今晚观测 M42”；“三角座”等歧义输入不会进入计算。
4. **重建 manifest**：从实际执行状态生成模型、约束、验证、人工确认、文件哈希和最终状态。
   验收：模板模式写 `model_used=false`；用户约束逐项可见；FAILED/EXPECTED_FAILURE/PASSED 有明确含义且由验证结果计算。

### 第 4 周补齐（建议随后 3-5 天）

1. 把 observation log 纳入统一 Schema，单独保存 `observation_log.csv/json`，并对缺失和矛盾字段做失败测试。
2. 完成 NL、outreach、review 全链路审计；chat 工具集覆盖 4 个核心 Skill，或明确将 chat 降级为演示入口而非总控。
3. 建立真正的 pytest 套件和一键验收命令，覆盖正常、不可见、歧义、强月光、时区、Qwen 失败和输出隔离。
4. 修订复盘证据等级：时间差只能证明“开始较晚”，除非日志明确写原因，否则不能直接归因为“团队迟到”。

### 科学验证与复现包（建议并行 2-4 天）

1. 固化可再分发的 SIMBAD 查询快照或提供联网重查 + 缓存流程，记录查询 ID、日期、字段定义、许可证和 SHA-256。
2. 让验证脚本真正覆盖坐标、视星等、角径、别名和类型；对 M24/M40 等特殊对象建立人工裁决表。
3. 在全新环境运行 `install -> validate -> 3 cases`，保存机器、依赖锁文件、命令和结果摘要。
4. 先更新 project plan 中的济南案例，再同步 transfer log 和 diff log，恢复单一事实源。

## 七、面向赛道方向的高收益增强建议

1. **做“天文实训证据图谱”，不是普通计划器**：每一句讲解、每个数字、每条复盘结论都带 `claim_id -> source/tool -> version -> human_confirmation`，现场点击即可追溯。这比增加更多目标更能体现 AI Ready Skills。
2. **把失败案例做成主角**：用同一个需求展示裸 Qwen、当前工具链和修复后的 fail-closed Skills；量化数值幻觉率、无来源事实率、失败识别率和人工核查时间。
3. **加入教学型反事实**：展示“提前/推迟 1 小时、换日期、换设备、月亮更亮”分别触发哪些约束，让学生先预测再查看确定性结果，突出“天文实训”而不只是活动生成。
4. **做一次 5-10 人真实校园小试验**：记录计划耗时、是否找到目标、能否解释推荐原因、遗漏项和下一轮变化。小样本但全流程的证据比华丽 UI 更有竞争力。
5. **建立一页可复现评分板**：固定 20-30 个正常/失败案例，显示科学误差、Schema 通过率、Qwen grounding、输出完整度和复跑一致性；答辩时一键运行。
6. **突出人在回路的可见性**：目标歧义、地点权限、弱证据归因、替换主目标和未验证事实必须出现确认卡，并保留确认前后 diff。
7. **暂缓复杂前端和太阳系扩展**：先把月球计算、证据链、失败封闭和复现包做到可当场审计；轻量 Streamlit 只负责展示这些证据，不承担新的业务逻辑。
