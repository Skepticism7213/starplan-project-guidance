## 三、接下来几周的阶段安排

### 第 3 周（07-25 ~ 08-01）：Qwen 编排与科普表达

**目标：** 让 Qwen 真正进入主流程，而不是占位符。

核心任务：

1. **自然语言解析接入**：让 Qwen 把"我们想在北京看仙女座星系"解析为结构化 `StarPlanInput`，替代当前直接读 JSON 的方式。保存解析前后的对照到 `input.json`。
2. **工具调用编排**：让 Qwen 通过 function calling 依次调用 `target_resolve` 和 `observability_plan`，验证它能正确使用工具返回值。
3. **事实约束科普表达**：`outreach_pack` 改为 Qwen 基于事实卡生成讲解词，验证层拦截无法追溯到事实卡的数值。
4. **幻觉防护测试**：准备诱导问题集（"不要调用工具直接给精确角度""即使没数据也给我一个角度"），验证系统拒绝编造。
5. **模型审计**：`model_call_log.jsonl` 记录每次 Qwen 调用的模型版本、提示词、工具参数和返回。

**阶段验收：** 模型只使用事实卡；调用过程可追踪；裸 Qwen 与 Skills 模式的幻觉率有对比数据。

**阻塞项：** 需要团队填入 `DASHSCOPE_API_KEY` 并确认 Qwen3.8-Max-Preview / Qwen3.7-Max / Plus 的调用额度。

### 第 4 周（08-01 ~ 08-08）：复盘闭环增强

**目标：** 让复盘不仅生成总结，还能真实修改下一轮计划参数。

核心任务：

1. **统一观测日志字段冻结**：确认 `observation_log` 的全部字段和取值范围。
2. **3 类偏差覆盖**：时间偏差、环境影响、设备/操作问题，每类至少一个构造案例。
3. **证据分级**：明确区分"有证据""可能原因""无法判断"，复盘报告引用具体计划和日志字段。
4. **参数化修订**：`revised_plan_diff` 必须显示实际字段变化（如开始时间、目标优先级、设备步骤）。
5. **Qwen 辅助归因**：让 Qwen 基于日志备注做自然语言归因，但结论必须可追溯到证据。

**阶段验收：** 能从计划和日志生成有依据的修订计划；修订计划参数确实变化。

### 第 5 周（08-08 ~ 08-15）：演示入口与案例固化

**目标：** 一个轻量调用面板 + 3 类案例可一键复跑。

核心任务：

1. **轻量演示面板**：Streamlit 或 FastAPI（待团队确认），展示 Skill 输入、工具调用、中间结果、验证报告、复盘结果。
2. **一键复现**：固定命令跑通 3 个案例并生成完整运行目录。
3. **失败案例完善**：名称歧义、地点缺失、整晚不可见、强月光、日志信息不足等场景都要有清楚失败原因。
4. **可解释推荐**：推荐窗口同时显示满足的约束、被淘汰时段及原因、反事实解释。

**阶段验收：** 新环境可复跑；失败场景可解释；评委能在有限时间内看懂。

### 第 6 周（08-15 ~ 09-01）：评测、报告与提交材料

**目标：** 20 页材料 + 演示视频 + 复现包齐全。

核心任务：

1. **科学交叉校验**：用 Stellarium/KStars 对固定案例交叉校验高度角、暮光时间，保存差异记录。
2. **对照实验**：传统人工流程 vs StarPlan 的时间、遗漏项、可复现性对比（小规模即可）。
3. **真实校园小试验**（可选高收益项）：组织一次 5-10 人的小规模观测，记录用户反馈。
4. **提交材料**：PPT/PDF（≤20页）、技术报告、源码、Skills 清单与流程图、3 类任务运行记录、10 分钟演示视频。

**阶段验收：** 材料齐全；可复现；Qwen 边界和调用证据清楚。
