## 1. 赛项背景

我们参加的是挑战杯“揭榜挂帅”阿里云榜题，方向为：

**赛道三方向三：星语·面向 AI 的天文实训**

赛项鼓励团队开发一个面向具体任务环节的天文 skills 包，从真实天文工作和科普实践中选择一个常用但尚未很好接入 AI 的环节，将其中的知识、工具、数据流程或科研方法整理成智能体可以稳定调用的天文 skills。

作品重点不是做泛用天文聊天助手，而是把原本分散、老旧或只适合人工操作的天文能力，改造成：

- 可运行
- 可检查
- 可复现
- 可复用
- 可被 AI 智能体调用的 AI Ready 组件

技术基础要求：

- 基座模型须基于 Qwen 系列模型。
- 开发平台需通过阿里云百炼平台调用，或采用比赛官网推荐的 QoderWork/Qoder/秒悟等工具调用系列模型。
- 允许围绕下游任务和领域数据做模型微调，但本项目暂不以微调为核心。
- 推荐搭建可交互前端、可调用测试 API，并制作 10 分钟内演示视频。

最终提交要求：

- 一份 PPT/PDF 文件，不超过 20 页。
- 可直接运行或加载到智能体的 skills 包，含依赖说明、安装方式和调用示例。
- 技能包流程图及 skills 清单，说明每个 skill 解决什么问题、何时触发、输入输出是什么。
- 至少 3 类典型任务的完整运行记录，保留关键中间结果、最终输出和人工校验入口。
- 若基于已有天文工具或开源项目改造，应说明来源、许可证、改造内容和智能体调用方式。
- 应用技术报告和源码。
- 附加提交：可交互前端页面或 10 分钟内演示视频。

评分标准：

- 科学价值 40%
  - 科学事实表达准确性
  - 内容转化、解释或展示的清晰度
  - 作品主题完整性与一致性
- 技术深度 30%
  - 模型、智能体、技能设计完整性
  - 多模态数据处理、生成、推理或交互能力
  - 结果校验、反馈迭代与稳定性设计
- 应用潜力 30%
  - 面向真实场景的使用价值
  - 作品演示、交互入口与交付完整度
  - 代码、结果与流程可复现性

## 2. 项目名称与定位

项目名称：

**星程 StarPlan Loop：面向 AI 的校园天文观测与科普实训闭环 Skills 包**

一句话简介：

**星程 StarPlan Loop 将目标解析、可观测性计算、科普活动设计、实际观测记录和偏差复盘封装为 Qwen 智能体可调用的 AI Ready Skills，让一次校园观测活动能够被计划、执行、检查并持续改进。**

200 字以内简介：

星程 StarPlan Loop 是面向 Qwen 智能体调用的校园天文观测与科普实训 Skills 包，聚焦“观测计划—活动执行—结果复盘”这一真实但分散的环节。用户输入目标、地点、日期、受众和设备后，系统结合 Qwen 与确定性天文工具生成计划和科普活动包；活动结束后导入观测日志，系统解释计划与实际的差异并生成下一次修订计划，同时保留中间结果与人工校验入口。

## 3. 为什么选择这个切口

真实天文工作流程大致为：

```text
提出目标
-> 查天体坐标/星表/文献
-> 判断可观测性
-> 制定观测计划
-> 申请/使用望远镜
-> 下载或采集数据
-> FITS 数据处理
-> 测光/光谱/可视化
-> 结果解释
```

真实天文科普流程大致为：

```text
选天象/主题
-> 判断当地是否可见
-> 设计观察路线
-> 生成星图/讲解词/安全提示
-> 组织公众观测
-> 收集照片/问题/反馈
-> 复盘与二次传播
```

我们选择切入中间这段，并把活动后的反馈纳入闭环：

```text
天体/天象选择
-> 本地可观测性计算
-> 最佳观测窗口
-> 科普活动包与设备建议
-> 实际观测日志
-> 偏差归因与下一次计划修订
-> 人工校验与复现记录
```

这个环节真实、高频、范围可控，适合本科生团队实现，也能展示“可运行、可检查、可复现、可复用”和“反馈确实影响下一轮计划”的特点。

## 4. 使用价值、现有痛点和改造必要性

### 使用价值

StarPlan 面向：

- 高校天文社团
- 中小学科普教师
- 天文馆/科技馆活动组织者
- 本科生天文实训
- 小型望远镜用户

解决的问题：

```text
我要观测/讲解某个天体或天象
-> 它在我这里什么时候可见？
-> 高度够不够？
-> 月光影响大不大？
-> 太阳是否已经落山？
-> 需要什么设备？
-> 公众能不能看懂？
-> 星图和讲解材料怎么做？
```

### 没有 StarPlan 时的传统流程

以“北京某高校天文社组织 M31 仙女座星系公众观测活动”为例，组织者需要：

1. 查 M31 的基础信息、坐标、视星等、角大小。
2. 将学校地点转换成经纬度和时区。
3. 查询日落、天文暮光、目标高度、方位角。
4. 判断月相、月亮高度、月亮和 M31 的角距离。
5. 手工挑选最佳观测时间。
6. 使用 Stellarium、KStars 或 Aladin Lite 制作星图或截图。
7. 编写活动流程、讲解词、设备清单和风险预案。
8. 人工核对地点、时间、目标是否正确。

痛点：

- 工具分散：目标数据、确定性计算、星图和文档工具各自独立；SIMBAD、JPL Horizons、Stellarium、KStars 等仅作为可选参考或交叉校验，不全部作为 MVP 依赖。
- 门槛高：新手不熟悉高度角、airmass、月相、天文暮光等概念。
- 容易出错：地点、时区、目标名称歧义、月光影响容易漏掉。
- 难复现：传统流程常只留下截图和文档，缺少中间数据。
- 难被 AI 调用：传统工具多面向人工操作，缺少统一输入输出和运行记录。
- 大模型容易幻觉：直接问模型“今晚能不能看某天体”，模型可能编造时间、方位、高度等数值。

### 改造必要性

观测计划不是简单问答，而是需要：

- 确定性天文计算
- 工具协同
- 数据来源追溯
- 科学表达转化
- 人工校验
- 中间结果留存

StarPlan 的核心改造逻辑：

```text
Qwen 负责理解和表达
专业天文工具负责计算和验证
skill 框架负责标准输入输出、中间结果和复现
```

## 5. 已有工具与我们的差异

已有工具：

- astroplan / Astropy：可做目标高度、airmass、可观测性计算。
- JPL Horizons：可查询行星、小行星、彗星等太阳系目标星历。
- Stellarium / KStars：可模拟星空和观测列表。
- Aladin Lite / WWT：可做交互星图与天文可视化。
- TOM Toolkit / StarWhisper Telescope：偏专业科研观测系统。

我们的定位不是替代这些工具，也不是首期全部接入它们，而是：

```text
把成熟天文工具链 AI Ready 化
```

传统工具流程：

```text
人打开软件
-> 人输入目标
-> 人看图/看表
-> 人截图
-> 人写活动方案
```

StarPlan 流程：

```text
AI 调用 skill
-> skill 调用确定性天文工具
-> 自动输出结构化结果、可选图表、讲解稿、观测日志复盘和校验报告
```

核心差异：

| 维度 | 传统工具 | StarPlan Loop |
|---|---|---|
| 使用方式 | 人手动点软件/写代码 | AI 或前端调用标准接口 |
| 输入 | 专业参数 | 自然语言任务 |
| 计算 | 工具各算各的 | 统一编排工具链 |
| 输出 | 图、表、截图 | JSON、CSV、可选图表、讲解稿、日志复盘、校验报告 |
| 科普转化 | 通常没有 | 自动生成活动流程和讲解词 |
| 复现 | 依赖用户记录 | 自动保存输入、中间结果、版本和日志 |
| AI 接入 | 基本没有 | 明确作为 Qwen 可调用 skill/API |

## 6. 核心技术路线

推荐系统结构：

```text
自然语言输入
-> Qwen/百炼解析任务
-> 标准化输入 schema
-> 目标解析
-> 可观测性计算
-> 科普活动包生成
-> 活动执行与观测日志导入
-> 偏差归因与修订计划
-> validation_report
-> 输出完整运行记录
```

专业计算原则：

```text
工具算
模型讲
报告验
人工确认
```

严禁让 Qwen 自己编：

- 高度角
- 方位角
- 月亮影响
- 日落时间
- 行星位置
- 星图

Qwen 只能基于工具生成的事实卡做任务理解、流程编排和科普表达。

## 7. Skills 清单

首期实现 4 个核心 Skill 和 1 个总控入口。星图和曲线属于展示层，不首期拆成独立 Skill；太阳系星历作为后续扩展。

| Skill | 解决问题 | 触发条件 | 输入 | 输出 |
|---|---|---|---|---|
| `target_resolve` | 把自然语言目标转成标准天体/坐标/类型 | 用户输入目标后 | 目标名、可选目标类型 | 标准名称、坐标、来源、置信度；歧义时要求确认 |
| `observability_plan` | 判断目标何时可观测并生成计划 | 目标和地点时间确定后 | 坐标、地点、日期、设备约束 | 可见窗口、高度/方位、airmass、暮光/月光风险、备选时段 |
| `outreach_pack` | 把确定性计算结果转成科普活动包 | 已有事实卡和计算结果后 | 事实卡、受众、设备 | 活动流程、讲解词、设备清单、安全提示、人工核对项 |
| `observation_review` | 对比计划和实际日志并修订下一轮计划 | 活动日志导入后 | 原计划、实际时间、结果、云量、设备、备注 | 偏差分类、证据、改进建议、修订后的计划 |

总控调用：

```python
from starplan_skills.runner import run_starplan

result = run_starplan(input_data)
```

总控入口负责识别用户意图、依次调用必要 Skill、保存运行目录，并同时返回机器可读结果和人类可读报告。

## 9. 三类典型任务

至少保证 3 类任务完整跑通。

### 案例 1：正常可观测活动

任务：北京高校天文社组织 M31 新生观测夜。

系统生成目标信息、推荐时段、高度曲线、设备建议和科普活动包，人工确认后形成活动计划。

### 案例 2：不适合观测及备选方案

用户输入的目标在指定时间高度过低、受月光影响或处于暮光阶段。系统必须展示触发的规则、计算依据和可替换的时间或目标，而不能只返回“不推荐”。

### 案例 3：实际活动复盘

导入一份包含迟到、云量增加、器材准备不足或目标选取不当的观测日志。系统区分可计算因素与人工记录因素，生成偏差报告并修订下一次活动计划。

若核心流程提前稳定，可增加土星观测作为太阳系扩展示例，但不得影响上述三类案例的完整性。

## 10. 验证方法

团队不是天文爱好者，也可以通过以下方法验证结果正确性：

```text
权威工具交叉验证 + 中间结果可追溯 + 规则校验 + 人工确认入口
```

验证对象：

- 目标解析是否正确
- 坐标来源是否可靠
- 地点和时区是否正确
- 高度角/方位角趋势是否合理
- 推荐窗口是否满足规则
- 科普稿是否忠实于事实卡

基础规则由团队在首周冻结并写入配置，不在代码和模型提示词中分散维护。初始规则至少覆盖：

- 目标高度、airmass、暮光和月光条件。
- 设备类型与目标可见性的基本匹配。
- 低高度、强月光和信息缺失的风险等级。
- 所有关键数值必须来自工具计算，不能来自大模型自由生成。

具体阈值需要用固定案例和参考工具校准后再冻结，不能把未经验证的经验阈值直接写成科学结论。

每次运行生成：

```text
validation_report.md
```

内容包括：

- 输入检查
- 目标检查
- 计算检查
- 工具版本
- 交叉验证记录
- 人工确认项

答辩表述：

> 我们团队并不依赖主观观星经验来判断结果正确性，而是采用“权威数据源 + 确定性计算 + 交叉验证 + 人工校验入口”的验证机制。目标信息和参考结果根据目标类型选择可靠来源，可见性由 Astropy/astroplan 等确定性工具计算；SIMBAD、JPL Horizons、Stellarium 或 KStars 可作为交叉校验，不作为首期核心服务的唯一依赖。大模型只负责语言理解、工具编排和科普表达，所有关键数值均来自工具输出，并在 validation_report 中保留来源、规则和中间结果。

## 15. 风险与控制

主要风险：

- 被认为只是套壳。
- 天文结果被质疑。
- Qwen 生成内容出现幻觉。
- 线上协作最后无法集成。
- 过度追求前端，忽略 skill 包本体。

控制方式：

- 用统一 schema 和标准 API 体现 skill 设计。
- 所有关键数值来自确定性天文工具；JPL Horizons 等在线服务仅作为按目标类型选择的可选校验。
- Qwen 只基于事实卡生成讲解稿。
- 每次运行保存 `calculation_manifest.json`、模型调用日志和人工确认记录。
- 每个案例保存完整中间结果和校验报告。
- 先完成本地计算最小链路，再接入 Qwen 和轻量展示层。
- 每周跑 3 个固定案例，其中至少一个包含失败或复盘场景。
