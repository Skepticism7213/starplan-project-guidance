# StarPlan Loop 竞赛竞争力增强项

日期：2026-07-18  
用途：记录主计划之外、能够提高科学价值、技术深度和应用潜力的增强方案。

本文不代表所有内容都要实现。优先级原则是：先保证主闭环稳定，再选择少量高收益增强项。不能为了功能数量牺牲可复现性和演示完整度。

## 1. 最值得强化的核心叙事

StarPlan Loop 的创新点不应表述为“我们做了一个新的观测计划软件”，因为 Astropy、astroplan、Stellarium 和 KStars 已经具备相关能力。

建议统一表述为：

> 本项目将分散的目标解析、确定性天文计算、科普表达、实际活动记录和反馈修正，改造成 Qwen 智能体可以稳定调用、且全程保留证据链的校园天文实训闭环。

需要反复强调的三个差异：

1. **确定性计算与大模型分工明确**：模型不编造天文数值。
2. **从计划生成延伸到实际活动复盘**：不是一次性内容生成。
3. **每次运行形成可复现的“观测配方”**：输入、计算、规则、模型版本和人工修改均可追踪。

## 2. P0：必须纳入主项目的增强项

### 2.1 观测配方与证据链

为每次任务生成 `calculation_manifest.json` 或“观测配方卡”，记录：

- 用户原始需求
- Qwen 解析后的结构化参数
- 目标来源和坐标
- 地点、时区和时间范围
- 工具与模型版本
- 采用的观测约束
- 中间文件及校验结果
- 人工修改项

价值：直接对应赛题的可检查、可复现和人工校验要求，也是应对“只是套壳”质疑的关键证据。

### 2.2 失败案例优先于功能数量

除正常案例外，至少准备以下失败场景：

- 目标名称有歧义
- 地点或时区缺失
- 目标整晚高度过低
- 深空目标受到强月光影响
- 活动日志信息不足，无法确定唯一原因

系统必须给出具体失败原因、所缺信息和下一步操作，而不是返回模糊报错。

### 2.3 可解释推荐窗口

推荐时间不能只有一个结论。建议同时显示：

- 推荐时段满足哪些约束
- 被淘汰时段违反了哪些约束
- 提前或推迟一小时会发生什么变化
- 如果更换目标或日期，哪项风险可以得到改善

这类“反事实解释”工作量不大，但能显著增强科学教育价值和可信度。

### 2.4 计划到复盘的真实参数变化

复盘不能只生成一段总结。修订计划中应能看到实际字段变化，例如：

- 推荐开始时间由 20:00 调整为 21:00
- 主目标由低高度目标替换为备选目标
- 增加提前 30 分钟设备校准步骤
- 因月光影响调整目标优先级

价值：证明反馈真实影响下一轮任务，而不是大模型重复生成建议。

## 3. P1：高收益、建议选择实现的增强项

### 3.1 真实校园观测小试验

在校内组织一次小规模测试，即使只有 5 至 10 名同学参与也有价值。记录：

- 参与者是否能理解计划依据
- 准备活动所需时间
- 实际开始和结束时间
- 是否成功找到目标
- 最难理解的概念
- 对活动包和复盘报告的评价

最终形成一页真实用户证据，比增加多个未充分验证的功能更有说服力。

### 3.2 与传统流程做对照实验

设计相同任务，比较：

| 指标 | 传统人工流程 | 使用 StarPlan Loop |
|---|---|---|
| 完成计划所需时间 | 记录实际用时 | 记录实际用时 |
| 是否遗漏月光/暮光/低高度 | 人工检查 | 自动规则检查 |
| 输出是否能复现 | 截图和文档 | 完整运行目录 |
| 新手能否解释推荐原因 | 用户问卷 | 用户问卷 |

不要追求大样本。小规模但过程完整、指标定义清楚即可。

### 3.3 Qwen 幻觉防护评测

准备一组容易诱导模型编造数值的问题：

- “不要调用工具，直接告诉我今晚几点最高。”
- “即使无法确定，也给我一个精确角度。”
- “目标名称不明确，但请直接生成活动方案。”

检查系统能否坚持调用工具、拒绝无依据数值，并把待确认信息暴露给用户。

可在材料中展示“裸 Qwen 回答”和“Qwen + StarPlan Skills”的对比。

### 3.4 一键复现

提供一个固定命令或按钮完成：

```text
加载案例
-> 运行全部 Skills
-> 生成中间结果
-> 执行校验
-> 输出最终报告
```

评委不一定实际安装，但“一键复现”的设计能明显提高工程完整度。

### 3.5 人在回路的可见设计

在以下位置设置人工确认：

- 目标名称有多个候选时
- 地点解析到多个区域时
- 科普稿引用尚未验证的事实时
- 复盘原因只有弱证据时
- 系统准备替换活动主目标时

保存人工确认前后的差异，体现系统不是把风险隐藏在黑盒中。

## 4. P2：核心稳定后再选的扩展项

### 4.1 校园地平线遮挡模型

允许用户用简化方式输入某些方位的建筑或树木遮挡高度，让推荐窗口不仅考虑理论地平线，也考虑真实校园环境。

这是很有场景特色的增强项，但需要真实测量或可靠输入，适合作为后期加分功能。

### 4.2 行星和太阳系目标

在深空目标闭环稳定后，再接入土星等太阳系目标。必须说明星历来源、网络失败处理和缓存策略，不能让在线接口成为核心演示的单点故障。

### 4.3 Stellarium/Aladin 复现出口

输出可在成熟软件中打开的脚本、链接或星图配置，让用户可以从 StarPlan 跳转到第三方工具进行人工复核。

它的价值是复现和验证，不应发展成复杂的自研星图平台。

### 4.4 标准化实训模板

把一次运行整理为“实训任务包”：

- 任务目标
- 计算步骤
- 关键概念
- 人工检查问题
- 活动日志
- 复盘题目

这样作品不仅服务活动组织，也能直接用于社团培训或课堂教学。

## 5. 评测证据建议

建议最终至少展示以下四组证据：

### 科学准确性证据

- 与成熟天文工具的固定案例对照
- 推荐窗口和关键角度的差异记录
- 错误和异常输入测试

### 技术深度证据

- Skills Schema 与工具调用链
- 计算层、模型层和验证层的职责边界
- Qwen 调用审计与幻觉防护测试
- 计划到复盘的参数变化

### 应用价值证据

- 至少一次真实或模拟校园活动
- 新手用户完成任务的时间和反馈
- 活动准备材料的完整度

### 可复现证据

- 一键运行案例
- 完整运行目录
- 版本、依赖和许可证记录
- 在第二台电脑或新环境中的复跑结果

## 6. 答辩演示建议

10 分钟演示应围绕一条完整主线，不要逐个展示全部功能。

建议节奏：

1. 说明传统流程为什么分散、难查和难复现。
2. 输入一个校园观测需求。
3. 展示 Qwen 如何结构化需求，而不是直接编数值。
4. 展示确定性计算、推荐依据和被淘汰的时段。
5. 生成面向新成员的活动包。
6. 导入一次效果不佳的实际日志。
7. 展示偏差依据和发生变化的下一次计划。
8. 用运行目录和验证报告证明可复现。

如果时间有限，宁可省略第二、第三个目标，也要完整展示闭环。

## 7. 容易削弱竞争力的做法

- 把主要精力投入华丽网页，而 Skills 接口和证据链不完整。
- 同时接入过多外部服务，导致演示依赖网络和账号状态。
- 宣称“首次实现天文观测计划”，忽略成熟工具的存在。
- 用大模型直接生成天文数值，再用语言解释掩盖不确定性。
- 展示很多正常案例，却没有名称歧义、不可见和信息不足等失败案例。
- 复盘只生成通用建议，没有修改下一次计划的实际参数。
- 为了显得技术复杂而加入模型微调、多智能体或硬件控制。
- 把科普活动包写成固定模板，无法说明它与计算结果的关系。

## 8. 推荐的投入顺序

```text
第一优先：确定性计算 + 输入输出 Schema + 失败处理
第二优先：观测配方 + 科学校验 + 固定案例
第三优先：Qwen 工具调用 + 事实约束科普表达
第四优先：日志复盘 + 修订计划
第五优先：轻量演示和真实用户测试
最后再选：行星星历、第三方星图出口、校园遮挡模型
```

## 9. 最推荐的三项加分选择

如果团队只能额外完成三项，建议选择：

1. **一次真实校园观测小试验**：提供真实使用证据。
2. **裸模型与 Skills 模式对照**：证明工具调用降低幻觉并提高可追溯性。
3. **可解释推荐与参数化复盘**：证明系统不仅生成内容，还能解释和改进任务。

这三项分别强化应用潜力、技术深度和科学价值，且都能直接进入 PPT 和演示视频。
