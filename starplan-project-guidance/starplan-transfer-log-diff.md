# Transfer Log 与 Project Plan 的差异清单

日期：2026-07-18  
对照基准：`starplan-loop-project-plan.md`  
已更新文件：`starplan-qoderwork-transfer-log.md`

本文只记录 transfer log 相对当前 project plan 的调整，方便团队审核；它不是第三份独立方案。

## 1. 已修改的核心差异

| 项目 | 原 transfer log | 当前 project plan | 处理结果 |
|---|---|---|---|
| 项目定位 | 观测前计划、星图和科普材料生成 | 观测计划—活动执行—日志复盘—下一次修订的闭环 | 已修改为 `StarPlan Loop` 闭环定位 |
| 一句话简介 | 生成观测计划与科普活动方案 | 让校园观测活动被计划、执行、检查并持续改进 | 已修改 |
| 核心流程 | 到人工校验与复现记录结束 | 增加实际日志、偏差归因和修订计划 | 已修改 |
| 核心 Skill 数量 | 5 个子 Skill：目标、可观测性、太阳系星历、星图、科普包 | 4 个核心 Skill：目标解析、可观测性计划、科普活动包、观测日志复盘 | 已修改 |
| Skill 命名 | `target_resolver`、`observability_planner` 等 | `target_resolve`、`observability_plan`、`outreach_pack`、`observation_review` | 已统一 |
| 图表功能 | `sky_visualizer` 作为独立 Skill | 图表是展示层，不在首期拆成独立 Skill | 已降级为展示层 |
| 太阳系星历 | 首期 Skill 和第二个必跑案例 | 核心稳定后再考虑的扩展 | 已后置 |
| 三类案例 | M31、土星、英仙座流星雨 | 正常可观测、不适合观测及备选、实际日志复盘 | 已改为围绕闭环的案例 |
| 第一阶段最低目标 | M31 生成计算、曲线、活动包和校验报告 | 在此基础上导入日志并生成复盘报告与修订计划 | 已增加闭环验收 |
| 阶段安排 | 第 3 周直接做土星和行星星历 | 第 3 周接 Qwen，第 4 周做日志复盘，第 5 周再做完整案例 | 已调整顺序 |

## 2. 保留但明确降级的内容

以下内容仍在 transfer log 中保留，但不再代表 MVP 必须完成：

- `JPL Horizons`：作为太阳系目标的后续星历来源或按目标类型进行交叉校验。
- `SIMBAD/astroquery`：作为目标信息参考来源；首期先保证目标目录和本地可复现链路。
- `Stellarium/KStars`：作为人工交叉验证和复现工具，不作为核心运行依赖。
- `Aladin Lite/WWT`：作为后续星图或可视化出口，不作为自研平台建设目标。
- 土星观测：核心闭环稳定后可加为扩展示例。
- 流星雨、月食等复杂天象：后续可以使用半结构化事件输入，但不进入首期闭环验收。
- 校园地平线遮挡模型：作为具有场景特色的后期加分项。

## 3. 新增或强化的内容

这些内容是原 transfer log 中不足、但当前 project plan 已明确的部分：

1. **`observation_review`**：读取实际观测日志，区分有证据的原因、可能原因和无法判断的情况。
2. **真实参数修订**：下一次计划必须修改时间、目标优先级、设备准备或流程步骤中的至少一项。
3. **`calculation_manifest.json`**：记录原始需求、结构化参数、目标来源、工具版本、约束规则、中间结果和人工确认。
4. **`model_call_log.jsonl`**：记录 Qwen 模型版本、提示词/工具调用和返回结果，证明阿里云能力的实际使用。
5. **失败场景验收**：增加目标歧义、地点/时区缺失、目标低空、强月光和日志信息不足等测试。
6. **核心计算离线复现**：在线服务不能成为核心演示的单点故障。
7. **轻量展示入口**：页面只负责调用 Skills 和展示证据，不以复杂前端为主要成果。

## 4. 仍然一致的内容

以下内容在两份方案中没有冲突，已保留：

- 参赛方向为赛道三方向三。
- 必须使用 Qwen/阿里云百炼等阿里云 AI 能力。
- Qwen 负责理解、编排和科普表达；确定性天文工具负责数值计算。
- 采用 Astropy/astroplan 等工具进行可观测性计算。
- 需要统一输入输出 Schema、Skills 清单、运行记录和验证报告。
- 需要提交源码、技术报告、PPT/PDF、演示视频和可运行/可加载的 Skills 包。
- 五人分工和六周推进节奏仍然适用，但任务顺序已按闭环重新排列。

## 5. 当前建议团队重点审核的三件事

1. 是否同意把“日志复盘和下一次修订”作为项目的主要差异化，而不是可选功能。
2. 首批目标目录是否先限定为固定、可审计的目标集合，避免一开始覆盖所有天体。
3. 是否接受“土星、流星雨、Stellarium/Aladin 等先后置”的范围控制。

只要这三项确认，transfer log 和 project plan 就可以作为同一套执行基准使用。
