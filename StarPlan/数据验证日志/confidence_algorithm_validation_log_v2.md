## 二、修复前发现的问题及修复措施

### 缺陷1: 别名遍历提前return（代码bug）

**修复前**: `_match_catalog_entry` 遍历别名时，遇到第一个包含匹配（0.85）就立即return，不再检查后续别名是否有精确匹配（0.95）。  
**影响**: 搜索"三角座"时，先命中"三角座星系"（包含→0.85），永远到不了"三角座"（精确→0.95）。  
**修复**: 改为遍历所有别名取最高分，仅精确匹配（0.95）可短路返回。  
**验证**: "三角座" → M33, confidence=0.95, requires_confirmation=False ✓

### 缺陷2: 确认阈值不考虑分差（设计缺陷）

**修复前**: 判定条件为 `best >= 0.9 AND second < 0.5`，不考虑best与second的差距。  
**影响**: "Dumbbell Nebula"精确匹配M27(0.95)，但M76"Little Dumbbell Nebula"得0.85，因0.85≥0.5触发歧义。同类影响5条用例。  
**修复**: 增加分差条件 `best >= 0.9 AND (second < 0.5 OR gap >= 0.08)`。  
**验证**: Dumbbell Nebula/Pinwheel Galaxy/武仙座球状星团/人马座球状星团 → 全部自动确认 ✓

### 缺陷3: 包含匹配对短查询过于宽泛（设计缺陷）

**修复前**: `norm_query in norm_alias` 无比例限制，2字符查询"Al"命中64个目标（"Sm**al**l...", "Andromeda G**al**axy"等）。  
**影响**: 候选列表被无关目标污染，用户真正想找的Altair/Aldebaran被挤出Top 10。  
**修复**: 增加比例限制 `len(query) >= len(alias) * 0.3`，查询长度不足别名30%时不算包含匹配。  
**验证**: "Al" → 候选列表正确显示Alpha Centauri/Altair/Aldebaran等（前缀匹配） ✓

### 缺陷4: 反向前缀匹配导致虚假结果（设计缺陷）

**修复前**: `norm_query.startswith(standard)` 条件使"M999"匹配到M9（"m9"是"m999"的前缀→0.3）。  
**影响**: 不存在的编号返回不相关结果。  
**修复**: 移除反向前缀匹配，仅保留正向（查询是标准名的前缀，如"M3"→"M31"）。  
**验证**: "M999" → confidence=0.0, standard_name="", requires_confirmation=True ✓

## 四、修复后算法行为确认

### 置信度打分规则（修复后）

| 匹配类型 | 分数 | 条件 |
|----------|------|------|
| 精确匹配标准名 | 1.0 | normalize后完全相等 |
| 精确匹配别名 | 0.95 | normalize后完全相等（短路返回） |
| 查询包含于别名 | 0.85 | query in alias 且 len(query) >= len(alias)*0.3 |
| 别名包含于查询 | 0.80 | alias in query 且 len(alias) >= 2 |
| 前缀匹配标准名 | 0.3 | standard.startswith(query) |
| 无匹配 | 0.0 | 以上均不满足 |

### 人工确认判定规则（修复后）

| 条件 | requires_confirmation |
|------|----------------------|
| 无匹配（confidence=0） | True |
| best >= 0.9 且 (second < 0.5 或 gap >= 0.08) | False |
| 其他情况 | True（附候选列表，最多10条） |

### 类型过滤

- 指定target_type且与条目不符时，score *= 0.5
- 空字符串type_hint等同于不过滤
